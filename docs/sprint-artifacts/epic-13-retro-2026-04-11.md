## Successes

1. **The app now has a cohesive, polished visual identity.** Every screen — home, card detail, add card, auth, settings, onboarding, sync indicators, and Apple Watch — was redesigned and implemented in a single sprint. The app looks and feels like a real product.

2. **Wave structure maximized throughput.** Foundation-first (Wave 1) unlocked parallel waves (2–4) without dependency conflicts. 12 implementation stories completed in 10 days.

3. **Design-to-implementation pipeline (Epic 12 → 13) proved its value end-to-end.** Figma frame references in story specs eliminated ambiguity. The review gate between design and implementation caught issues early (icon system, interaction patterns).

4. **Party mode for story development produced better outcomes.** Multi-agent collaboration during story refinement and problem-solving sessions consistently surfaced edge cases and design nuances that solo work would miss.

5. **QA and code review loops caught real issues before merge.** Touch targets, accessibility, dark mode gaps, hook ordering violations, NativeWind gotchas — all caught and fixed in the review cycle, not in production.

6. **Zero production incidents** despite 272 files changed and ~20K lines added.

---

## Challenges

1. **NativeWind caused persistent implementation friction.** ~23% of all commits were fix/rework commits, with NativeWind-related issues appearing across multiple stories: Pressable style callbacks, scroll behavior, conditional layout logic, dark mode inconsistencies. The library adds a translation layer that creates unpredictable interactions with React Native's layout system.

2. **Complex interactive screens (Card Detail, Add Card) had the most rework cycles.** These screens involve scroll behavior, gesture interactions, and multi-step flows that static Figma frames don't fully capture. 13-3 had barcode positioning issues; 13-4 had route bridge coverage gaps.

3. **Spec-to-implementation gaps still exist for dynamic behavior.** Figma specs tell you _what_ the screen looks like but don't always capture how scroll, animation, and conditional layouts behave under different states.

## Key Decisions

### DEC-S11-RETRO-001: NativeWind Evaluation

**Date:** 2026-04-11
**Context:** NativeWind was responsible for a disproportionate share of fix commits (~23%). The `Pressable` style callback restriction in AGENTS.md exists specifically because of NativeWind interference. Dark mode, scroll interactions, and conditional layouts repeatedly caused issues.
**Decision:** Evaluate NativeWind alternatives or mitigation strategies before the next UI-heavy epic. Options include: (a) migrating to `StyleSheet` + design tokens, (b) using Unistyles or Tamagui, (c) establishing stricter NativeWind patterns/guardrails.
**Status:** Open — to be resolved before next design implementation sprint.

### DEC-S11-RETRO-002: Party Mode as Default for Story Development

**Date:** 2026-04-11
**Context:** Collaborative party mode sessions during Sprint 10 (design) and Sprint 11 (implementation) consistently produced higher-quality decisions and caught more edge cases than solo work.
**Decision:** Party mode is now the **default** for all story refinement and complex problem-solving. Solo mode only for trivial/mechanical tasks.
**Status:** Approved — effective immediately.

### DEC-S11-RETRO-003: Definition of Ready & Definition of Done

**Date:** 2026-04-11
**Context:** Sprint 11 revealed that the quality of story execution depends heavily on the quality of story preparation. Some stories were well-defined with clear Figma references and interaction specs; others required mid-flight discovery. A formal gate system will reduce rework and improve predictability.
**Decision:** Adopt formal Definition of Ready (DoR) and Definition of Done (DoD) as mandatory process gates. See full definitions below.
**Status:** Approved — effective Sprint 12.

---

## Definition of Ready (DoR)

A story is **ready for development** only when ALL of the following are true:

| #   | Gate                   | Description                                                                                                                           |
| --- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Design Approved**    | Figma frames exist and are approved by stakeholder (ifero). Frame names documented in the story.                                      |
| 2   | **Story Spec Final**   | Story file created via Create Story workflow (party mode). All acceptance criteria defined, numbered, and testable.                   |
| 3   | **Interaction Spec**   | For complex screens (multi-step flows, scroll behavior, animations): interaction behavior is documented — not just the static visual. |
| 4   | **Dependencies Clear** | All blocking stories are `done`. Required APIs, data schemas, or components are available.                                            |
| 5   | **Edge Cases Defined** | Empty states, error states, offline behavior, and dark mode are explicitly addressed in ACs or design frames.                         |
| 6   | **Tech Notes**         | Any known technical constraints, library quirks (e.g., NativeWind gotchas), or architectural decisions are listed.                    |
| 7   | **Testability**        | QA knows how to verify each AC. Test IDs, manual test steps, or automation approach defined where relevant.                           |

**Gate keeper:** Story cannot move from `drafted` → `ready-for-dev` without passing all 7 gates.

---

## Definition of Done (DoD)

A story is **done** only when ALL of the following are true:

| #   | Gate                       | Description                                                                                              |
| --- | -------------------------- | -------------------------------------------------------------------------------------------------------- |
| 1   | **All ACs Pass**           | Every acceptance criterion is implemented and verified.                                                  |
| 2   | **Code Review Approved**   | Dev agent code review completed with `APPROVED` status. All feedback addressed.                          |
| 3   | **QA Review Approved**     | QA agent review completed. Accessibility, dark mode, edge cases, and error states verified.              |
| 4   | **Tests Pass**             | All existing tests pass. New tests added for new functionality. No test regressions.                     |
| 5   | **Lint & Typecheck Clean** | `npx expo lint` and TypeScript compilation pass with zero errors.                                        |
| 6   | **PR Merged**              | Pull request created, reviewed, approved by stakeholder (ifero), and merged to main.                     |
| 7   | **Sprint Status Updated**  | Story status is `done` in `sprint-status.yaml`.                                                          |
| 8   | **No Known Defects**       | No open bugs or regressions introduced by the story. If discovered later, a follow-up ticket is created. |
