## Successes

1. **Critical pipeline work is finished.** Both planned stories for Sprint 12 are done, including the watchOS TestFlight pipeline and the CI/CD documentation that reflects the current implementation.

2. **WatchOS distribution was successfully integrated into the iOS archive.** The watchOS app is now embedded via `@bacons/apple-targets` instead of a separate outdated `watch-ios/` Xcode project.

3. **Documentation caught up with implementation.** `docs/cicd.md` was updated to remove outdated claims, add watchOS CI/CD details, and clarify the release runbook and provisioning path.

4. **Priority gating worked well.** The watchOS build process was treated as a blocker story, which helped keep the sprint focused on the real delivery outcome: a single TestFlight archive for iOS + watchOS.


## Challenges

1. **WatchOS distribution complexity remains a risk.** Apple requires the watchOS app to be embedded inside the iOS archive, and Expo prebuild can be brittle when the native project is modified. This adds overhead to every change in the iOS/Watch target configuration.

2. **Documentation is only as good as the implementation it follows.** The old CICD doc contained inaccurate claims about notifications and rollback behavior; we needed to clean those up before the next release.

3. **Legacy path cleanup is important.** The old `watch-ios/` project and related references had to be removed carefully to avoid stale configuration or tests pointing to the wrong location.


## Key Learnings

- **Treat watchOS as part of the iOS delivery pipeline, not a separate product.** Once the target is embedded correctly, the archive workflow becomes the single source of truth.

- **Keep CI/CD documentation in sync with implementation.** If the docs drift, new contributors will follow an incorrect build process or assume unsupported features exist.

- **Blocker stories should be explicit.** Story 11-6 was clearly labeled P0 and blocked the real-device delivery outcome, which helped maintain focus.
